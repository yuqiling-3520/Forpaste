# 技术栈迁移重构计划（Godot → Tauri）

> 本文件为计划稿。沿用本项目既有方法论：先出计划，用户确认前不执行任何实现代码。
> 本稿目标是确立「大致开发框架 + 分阶段计划」，细节方案在每个阶段开始前另出子计划。

---

## 1. 文档目的

把当前基于 Godot 4.6.2 的剪切板工具，迁移重构到更适合本类「系统托盘小工具 + 跨平台 + 大模型协作开发」的技术栈（Tauri 2）。

本次重构要同时达成三个目标：

1. 根治原项目反复爆发的「窗口控制类」技术难题。
2. 大幅降低代码负担与软件体积（178MB → 个位数 MB）。
3. 让工程结构更适合大模型开发，且重要功能遵守组件模式，便于大模型独立恢复 / 重写单个组件。

本稿不立即改造代码，仅确立框架与阶段。

---

## 2. 背景与根因结论

### 2.1 原项目的真正负担来源

通读原项目 7 份「修复计划」后，结论是：它们不是 7 个独立问题，而是同一个结构性矛盾的反复爆发——

> 应用的「系统集成层」与「Godot 引擎」边界摩擦：每个原生能力（剪切板 / 热键 / 边界缩放 / 主窗口隐藏）各自获取 HWND、各自替换 WndProc、各自处理「句柄就绪时机」，导致同一类问题在每个桥接里反复出现、反复打补丁。

典型表现：

```text
主窗口隐藏：Godot 不让普通 hide() 主窗口
边界缩放：缺 WS_THICKFRAME，要手处理 NCCALCSIZE/NCPAINT/NCACTIVATE
窗口离屏：多屏坐标失效，Godot 不理多屏
滚动卡顿：Godot Control 节点树不适合长列表
```

### 2.2 为什么换栈能根治

Tauri 的窗口是**真正的操作系统原生窗口**，不是引擎视口。因此原项目最痛的一整类难题，在新栈里**从设计上消失**，而不是靠补丁缓解：

| 原难题 | 在 Tauri 下的状态 |
|---|---|
| 主窗口隐藏报错 | `window.hide()/show()` 任意窗口一等公民，直接可用 |
| 边界缩放 / 窗口样式 | `decorations:false` + `resizable:true` = 无边框且原生可缩放，无需碰 WndProc |
| 窗口离屏修正 | 官方 `tauri-plugin-window-state` 自带保存/恢复/离屏处理 |
| 句柄就绪时机 / 桥接重挂接 | 不存在该问题：不需要自己抢 HWND |
| 滚动卡顿 | 浏览器对长列表渲染高度成熟，虚拟列表库现成 |

**剪切板监听、全局快捷键、开机自启、单实例**——这四项原本占用大量 Win32 桥接代码的能力，在 Tauri 全部是**官方插件**，一行注册即可，大模型无需再写一行 Win32。

### 2.3 真正剩下的「平台专属」能力

诚实评估，迁移后真正还需要平台专属代码的，只剩两项：

```text
跟随插入符：获取系统文本光标屏幕坐标
十键粘贴：向目标窗口模拟 Ctrl+V
```

这两项收进「平台移植层」，正好承接原项目《主窗口隐藏桥接修复计划 4.3 平台差异隔离》《便利贴化改造计划书 4.11 平台桥接组件》已命名的方向：Windows 先实现，macOS/Linux 留填空骨架。

## 3. 技术栈选型结论

| 维度 | 结论 |
|---|---|
| 框架 | **Tauri 2**（已稳定，本项目所需系统能力均有官方插件） |
| 后端语言 | Rust（仅写插件注册 + 少量胶水 command，业务不在这层） |
| 前端语言 | TypeScript（强类型，拦截大模型幻觉） |
| 前端 UI 框架 | **待定**：Svelte / React 二选一（见 6.1 决策项） |
| 体积 | 3–10 MB（系统 WebView，不打包浏览器） |
| 内存 | 约 40–120 MB（借系统内核，远低于 Electron） |
| 跨平台 | 一套前端代码，三系统各用系统 WebView 渲染 |

选型核心理由（对齐「大模型友好」第一标准）：

1. **UI 用 Web 技术 = 大模型训练语料最丰富的领域**，改 UI 几乎不出错。
2. **系统能力插件化 = 大模型不必再碰 Win32**，根除原项目整类难题。
3. **前端组件 + 后端 command 的切割粒度最细**，单组件可独立重写不牵连其他。

已知代价（如实记录）：

```text
Linux 的 WebKitGTK 渲染偶有差异，需在 Linux 实机实测（交未来贡献者）。
Windows 7/8 需额外装 WebView2（Win10/11 自带，目标用户几乎无影响）。
引入 Rust，但需手写的 Rust 极少且模式固定。
```

## 4. 总体架构框架

采用分层 + 组件模式，每层职责单一，组件可独立重写。

```text
┌─────────────────────────────────────────────┐
│ UI 组件层 (前端 / TypeScript)                  │
│   历史列表 / 记录行 / 搜索栏 / 设置面板          │
│   状态消息 / 主题 —— 每个组件一个文件            │
├─────────────────────────────────────────────┤
│ 前端协调层 (组合根 + 状态管理)                  │
│   应用状态 store / 主协调逻辑 / 业务编排         │
├─────────────────────────────────────────────┤
│ IPC 契约层 (强类型边界)                        │
│   前端↔后端的 command 与事件，集中定义类型       │
├─────────────────────────────────────────────┤
│ 后端能力层 (Rust)                              │
│   剪切板服务 / 快捷键服务 / 窗口服务             │
│   存储服务 / 托盘服务 / 粘贴服务                 │
│   —— 大多是官方插件注册 + 薄胶水                 │
├─────────────────────────────────────────────┤
│ 平台移植层 (仅平台专属)                         │
│   跟随插入符 / 粘贴模拟                          │
│   windows(先做) / macos(空壳) / linux(空壳)     │
└─────────────────────────────────────────────┘
```

### 4.1 组件模式原则（继承原项目）

```text
前端协调层只做编排，不写底层细节（对应原「主脚本只做组合根」）。
子组件通过事件 / store 通信，不互相直接操作。
UI 组件不直接读写存储，存储集中在后端存储服务。
平台能力与 UI 解耦，UI 只调 IPC 契约，不知道平台细节。
单个组件单一职责，可被大模型整体重写而不影响他人。
```

### 4.2 能力到实现的映射表

| 能力 | 实现方式 | 备注 |
|---|---|---|
| 剪切板监听 | 剪切板插件 + 监听事件 | 监听能力选型见 6.1 |
| 剪切板读写 | `tauri-plugin-clipboard-manager` | 官方 |
| 全局快捷键 | `tauri-plugin-global-shortcut` | 官方 |
| 开机自启 | `tauri-plugin-autostart` | 官方 |
| 单实例唤起 | `tauri-plugin-single-instance` | 官方，须首个注册 |
| 窗口位置持久化/离屏 | `tauri-plugin-window-state` | 官方，根治离屏难题 |
| 本地存储(历史/设置) | `tauri-plugin-store` 或 fs | JSON，平移原数据结构 |
| 无边框 + 可缩放 + 拖拽 | Tauri 原生窗口 API | 根治窗口样式难题 |
| 托盘菜单 | Tauri 内置 tray | 显示/隐藏/置顶/退出 |
| 主题/缩放/搜索/虚拟列表 | 纯前端 | 大模型主场 |
| 跟随插入符 | 平台移植层 | Windows 先做 |
| 十键粘贴(模拟 Ctrl+V) | 平台移植层 / `enigo` | Windows 先做 |

## 6. 决策项（已确认 2026-06-14）

### 6.1 前端 UI 框架 —— ✅ 选定 React

```text
结论：React。
理由：训练语料最丰富，大模型最熟、写 UI 出错率最低，生态最大。
代价：产物体积略大于 Svelte，但仍在个位数 MB 目标内，可接受。
```

### 6.2 剪切板监听插件 —— ✅ 选定「社区插件（依赖）」

```text
结论：引入社区 tauri-plugin-clipboard 作为依赖，使用其内置「内容变化监听」。
机制说明：插件 = 通过 Cargo.toml / package.json 声明的依赖（下载植入），
         源码不进本仓库；我们只声明 + 注册，不重写插件本体。
理由：监听能力最完善、省去自写轮询胶水，符合「大模型不碰底层」第一标准。
代价：多依赖一个第三方社区插件（高人气，风险可控）。
```

### 6.3 数据迁移 —— ✅ 选定「不迁旧版，但内建版本化迁移机制」

```text
结论：
  1. 不迁移旧 Godot 版的 user://clipboard_history.json（不做一次性导入）。
  2. 但从第一天起，存储层必须内建「版本化数据迁移」能力，
     为未来版本更新时的数据结构升级预留平滑迁移管线。
落地要求（写入阶段 1 子计划）：
  - 所有持久化数据带 schema_version 字段。
  - 存储服务读取时检测版本，低于当前版本则按迁移链逐级升级后再使用。
  - 迁移失败要有兜底（备份原文件 + 降级提示，不静默丢数据）。
理由：放弃历史 Godot 数据可简化迁移；但新栈自身的长期演进必须可平滑升级，
     避免重蹈「改结构即丢用户数据」的覆辙。
```

## 7. 分阶段计划（大致）

> 每阶段开始前另出子计划稿；每阶段结束做验收 + 记录，符合原项目阶段执行记录惯例。

### 阶段 0：技术栈选型确认与脚手架
- 确认 6.1/6.2/6.3 决策项。
- 初始化 Tauri + 前端框架空项目，跑通空窗口。
- 验收：能启动一个无边框透明窗口。

### 阶段 1：数据层与存储服务
- 平移原 `剪切板存储` / `应用设置存储` 的数据结构与逻辑到后端存储服务。
- 实现历史/设置的读写、防抖保存、最大条数裁剪。
- **内建版本化迁移机制（决策 6.3）**：数据带 `schema_version`，读取时检测并按迁移链升级，失败有备份兜底。
- 不导入旧 Godot 版 JSON（决策 6.3）。
- 验收：历史与设置可持久化、可恢复；写入一个旧 schema 测试文件能被正确升级。

### 阶段 2：核心 UI 与剪切板主流程
- 实现历史列表（虚拟列表）、记录行、搜索栏、状态消息、主题。
- 接入剪切板监听 → 记录 → 列表刷新 → 点击复制回写。
- 验收：复制内容能自动入库、可搜索、可点击复制、可删除/清空。

### 阶段 3：窗口形态与桌面壳
- 无边框 + 可缩放 + 顶部拖拽（原生，不碰 WndProc）。
- 托盘菜单（显示/隐藏/置顶/退出）。
- 单实例唤起。
- 窗口位置持久化 + 离屏恢复（官方插件）。
- 验收：原「窗口难题清单」逐项通过且无需补丁。

### 阶段 4：全局能力
- 全局快捷键唤醒（可自定义）。
- 开机自启。
- 卡片缩放（Ctrl + / - / 0）。
- 验收：快捷键、自启、缩放行为与原版对齐。

### 阶段 5：平台移植层
- 定义跟随插入符 / 十键粘贴的平台契约。
- Windows 实现；macOS/Linux 空壳 + README + 能力降级。
- 验收：Windows 全功能；其他平台可编译可运行（缺能力优雅提示）。

### 阶段 6：跨平台扩展文档与发布整理
- 写《平台移植贡献指南》：如何为新平台填空。
- 图标、版本号、打包配置、三平台产物。
- 验收：贡献者能照文档新增一个平台实现。

## 8. 风险点

```text
8.1 Rust 学习成本：用插件化把手写 Rust 降到最小；胶水代码模式固定。
8.2 Linux 渲染差异：UI 简单，影响有限；交未来 Linux 贡献者实测。
8.3 剪切板监听插件选型：阶段 0 先做小验证，确认监听稳定性。
8.4 跟随插入符/粘贴模拟：唯二平台专属，单独子计划、单独验证。
8.5 数据迁移：旧 JSON 导入要做好字段校验与失败兜底。
8.6 开发者只有 Windows 机器：macOS/Linux 仅保证可编译 + 降级，不保证体验。
```

软件体积显著下降到个位数 MB 级别。
